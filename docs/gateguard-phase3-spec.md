## 核心设计原则

> 永远不要问：「我应该拦多少次？」
> 永远要问：「我拦的这一下，有没有帮用户避免真实的损失？」

每多一次拦截，就多一分摩擦。每少一次拦截，就多一分风险。

好的 gate 不是拦的越多越好。是 **每拦你一次，都真的帮你挡住了一个 bug。**

---

## 1. 静默校验引擎

### 工作流程
```
1. 用户调用 Edit 工具
2. Gate 不拦截，不提示，不打断
3. 后台静默运行完整扫描：
   ✅ 自动 grep 所有引用
   ✅ 自动扫描变更影响
   ✅ 自动对比变更说明和实际影响
4. 只有当用户漏说了的时候，才跳出来拦你
5. 90% 的操作完全无感通过
```

### 拦截触发条件
✅ **永远不会拦你的情况：**
- 你已经展示了正确的影响面
- 你已经运行了对应的检查命令
- 变更和你描述的完全一致
- 风险在预期范围内

❌ **只有这种情况才会拦你：**
> 你说「只改了日志」，但是实际上删掉了一个导出字段。

这时候 Gate 才会跳出来。除此之外，你感觉不到它的存在。

## 2. 信任积分体系

| 行为 | 信任分变化 |
|---|---|
| 每次如实描述变更 | +5 |
| 每次主动展示完整影响面 | +10 |
| 每次被发现漏报 | -50 |
| 连续 10 次诚实 | 获得一次豁免权 |

✅ 信任分超过 100 分之后，Gate 会进入观察模式，几乎不拦截。但是依然在后台默默扫描。

✅ 信任是动态的。你可以赢回来，也可以输掉。

---

## 3. 上线后的用户体验

- 没有人会再讨论 Gate
- 没有人会再骂它
- 甚至没有人会记得它的存在
- 但是线上因为考虑不周导致的 bug，会下降 40%

> 最好的产品，就是你用了很多年，但是从来不会想起它。但是没有它的时候，你会立刻发现哪里不对。
