# GateGuard 第二阶段落地规范 v1.0

> 发布日期: 2026-04-19
> 负责人: P8
> 核心突破: 从「文件门控」→「变更影响面门控」

---

## 底层逻辑

> 你改了什么，我就检查什么。别的我不管。

当前 Gate 最大的 bug：你改了文件里的一行注释，它会拦你。你改了一个函数签名，它还是拦你。在它眼里，所有修改都是一样的。

这一阶之后，Gate 不再看你改了哪个文件，它看你**改了什么东西**。

---

## 1. 增量变更分析引擎

### 接口定义
```typescript
interface ChangeImpact {
  change_type: 'function_signature' | 'function_body' | 'variable' | 'export' | 'import' | 'comment' | 'whitespace';
  modified_symbol: string | null;
  call_sites: Array<string>;
  reference_sites: Array<string>;
  risk_factors: Array<string>;
  impact_score: number;
}
```

### 分析规则
| 变更类型 | 处理策略 | 检查动作 |
|---|---|---|
| 注释 / 空格 / 日志 | ✅ 直接放行 | 什么都不做 |
| 函数内部逻辑修改 | 🟡 轻度检查 | 查这个函数的调用方数量 |
| 函数签名修改 | 🟠 中度检查 | 自动 grep 所有调用点 |
| 删除导出变量/函数 | 🔴 强制检查 | 列出所有引用这个符号的文件 |
| 新增导出 | 🟡 轻度检查 | 记录新增 API |
| 新增 import | ⚠️ 观察 | 记录新增依赖 |

---

## 2. 上下文感知门控逻辑

### 拦截判定流程
```
1. 用户调用 Edit 工具
2. 提取 diff，运行增量分析
3. 识别变更的符号和影响面
4. 计算影响分，而不是文件分
5. 只拦截变更带来的新风险
6. 之前已经通过验证的部分，永远不再拦
```

### 核心优化点
✅ 同一个文件，改注释直接过，改函数签名才拦
✅ 同一个函数，第一次改拦你，第二次改就不拦了
✅ 你已经列出过调用点，后面再改就不用再列一遍
✅ 你说「我已经检查过所有调用点」，Gate 会真的去验证你有没有搜过

---

## 3. 事实自动校验

> 你说什么不重要，你做了什么才重要。

| 用户说 | Gate 后台动作 | 结果 |
|---|---|---|
| 「我检查过了」 | 查会话历史有没有跑过 Grep 搜索这个符号 | ❌ 没跑过 = 不通过 |
| 「没有其他调用点」 | 自动后台跑一次 grep 验证 | ✅ 真没有 = 放行 |
| 「这只是改注释」 | 自动 diff 分析是不是真的只改了注释 | ✅ 属实 = 直接放行 |

✅ 这一阶上线之后，再也不会有人说「我就改了个注释你拦我干嘛」。
